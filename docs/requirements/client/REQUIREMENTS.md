# FreeChat 客户端要求

## 客户端职责边界

客户端负责：

- 交互展示。
- 输入约束。
- 权限状态提示。
- 响应式布局。
- 轻量预校验和错误提示。

客户端不负责：

- 最终权限裁决。
- 计费免费额度判断。
- 核心业务规则兜底。
- 审计归属确认。

所有安全相关判断必须由服务端再次校验。

## 移动端兼容

- 所有 UI 必须响应式适配手机端。
- 房间主界面应保持干净，高频协作入口留在主导航；排障、设置、执行记录等放在右侧设置或移动覆盖层。
- 设置操作尽量在当前上下文完成，减少不必要二级跳转。

## 权限 UX

- viewer/editor/owner 的能力差异要在 UI 上明确提示。
- 不可用操作可以隐藏或禁用，但不能把这种处理当作安全边界。
- 后端返回 403/404/validation 错误时，客户端要友好提示，不吞掉错误。
- Agent 模型配置 UI 必须明确费用来源：平台模型可能产生平台模型费；我的私有模型使用自己的 API Key，不收平台模型费；共享/市场模型按提供者定价结算。
- 通讯录 Agent 详情页是 Agent 默认模型主入口；设置后对所有房间、私聊、工作组入口统一生效。房间成员列表只展示“继承通讯录默认/当前房间覆盖”，并允许项目 owner/editor 设置覆盖或恢复继承。

## 文件上传与文件面板

- Agent 知识库上传使用专用 multipart API，不在浏览器端用 `file.text()` 伪装 JSON 创建；上传后要清空 file input value，允许连续选择同一个知识文件。只允许文本类知识文件，二进制文档需提示用户先转换。
- Agent 运行界面和运行说明应强调知识库是按需渐进式加载：Agent 先 `knowledge search`，再 `knowledge read` 少量命中内容；不要把全部知识库一次性塞入上下文。
- 文件面板的上传按钮必须绑定实际上传 action；如果上层传入 `uploadLocalFile` 而不是 `onUpload`，组件也必须能正常上传。
- 上传后要清空 file input value，允许连续选择同一个文件再次上传。
- 删除/打开等操作应传递明确的 `path`，避免把整个 FileNode 当成路径导致 `[object Object]` 请求。
- 房间聊天输入区自带附件上传必须提交 `content`、`mentions`、`reply_to` 和 `files`；有附件且 @ Agent 时，前端仍要把 mentions 放入 multipart，避免附件消息不能触发指定 Agent。
- 客户端可做空路径/非法路径提示，但最终路径清洗、权限和写入仍由服务端校验。

## 前端架构拆分

- 普通代码文件目标控制在 1000 行以内。
- 页面组件尽量负责组合和渲染。
- 复杂状态、动作和副作用应抽到：
  - controller hook。
  - action factory。
  - reducer/handler map。
  - API core/client 模块。
- API 客户端保持聚合导出兼容时，底层请求、token、错误解析、日志等横切逻辑应集中在 core 模块。
- 大页面和设置面板应按需懒加载，控制移动端首屏体积。

## Agent CLI / 小蜜界面代操作

- Agent Client 每次运行前应同步服务端 runtime spec 和 CLI，使 `./freechat app list/help/call`、`tool call`、`raw` 始终与服务端 App Action Registry 一致。
- 小蜜代替界面操作时，前端展示仍以服务端结果和确认卡为准；删除、成员角色、邀请、敏感模型/知识库写操作不能仅凭前端隐藏按钮作为安全边界。
- 用户上传本地文件仍由前端 file input/聊天附件完成；小蜜可操作已上传的文件引用或 Agent 知识库文本内容。
- 小蜜代替界面操作时，产品语义是“小蜜与当前用户同权”：小蜜可代用户操作其有权访问的 Agent、账单、知识库和项目；如果涉及目标项目/房间，应把目标 roomId 传给服务端并由服务端校验。
